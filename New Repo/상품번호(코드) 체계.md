# 상품번호(코드) 체계

## 1. 설명

- 상품번호 체계는 주문을 유일하게 식별하기 위한 규칙
- 카테고리 + 일련번호 + 옵션1 + 옵션2

주문번호 체계는 주문을 유일하게 식별하기 위한 규칙입니다.

* 품번코드 = 카테고리 코드 + 일련번호
* 단품코드 = 옵션1 + 옵션2
* 상품코드 = 1. 품번코드 + 2. 상품코드

| 구성 요소 | 예시              | 설명                 |
| -----    | --------------- | ------------------ |
| 카테고리 코드 | CLO1010     | 의류>상의>티셔츠  |
| 일련번호  | 9999          | 해당 카테고리내 상품 순번 |
| 옵션1+옵션2  | 999         | 옵션값 분류를 위한 순번|
| 합친 형태 | CLO1010-9999-160 | 구분값을 넣어줌     |

## 2. '잘 설계된' 상품 코드가 중요한가?
### 1. 상품 코드 부재 시 발생하는 문제점
- 운영 관리의 한계 발생
    - 상품 수 기하급수적 증가 시 엑셀/수기 관리의 한계 직면
    - 재고 관리, 주문 처리, 데이터 분석에 큰 어려움 초래
- 재고/배송 문제 심화
    - 오배송이 잦아짐
    - 잘 팔리는 상품 파악 미흡으로 재고가 쌓이는 악순환 발생
### 2. 체계적인 '상품 코드' 설계의 중요성
   - 데이터 연결의 핵심 역할 수행
      - 이커머스 생태계의 모든 데이터를 연결하는 핵심 '식별자(Primary Key)' 역할
   - 비즈니스 효율성 확보:
        -  비즈니스의 확장성과 효율성을 결정하는 설계 영역
   - 구체적인 효과:
        -  재고 관리의 정확성 증대
        -  데이터 분석을 통한 비즈니스 인사이트 제공
        -  궁극적으로 자동화의 기반 마련
### 3. 상품코드의 목적
  - 관점: 개발자 및 시스템 기획자의 관점에서 상품 코드의 본질 파악
  - 분석 대상: 국내 대표 쇼핑몰의 코드 시스템 심층 분석 (설계 철학 파악)
  - 제시 목표: 견고한 자체 상품 코드 시스템을 구축하는 실용적인 가이드라인 제시

## 단순한 식별자를 넘어: 좋은 코드 설계의 핵심 원칙
### 상품 코드의 종류
-  내부 관리용 코드 (SKU: Stock Keeping Unit)
    - 정의
      - 판매자가 자체 재고 관리를 위해 상품의 가장 작은 단위(예: 색상, 사이즈별)에 부여하는 고유 식별자
      - SKU는 판매자 내부적으로 생성하고 관리하는 코드로, 비즈니스 요구에 맞게 자유롭게 설계할 수 있음
    - 목적: 
      - 재고 추적, 판매 분석, 창고 관리(WMS) 및 전사적자원관리(ERP) 시스템 연동의 핵심
      - 핵심 주제는 바로 이 SKU 설계
-  국제 표준 상품 코드 (바코드: GTIN, EAN 등)
    - 정의 
      - GS1 국제기구가 관리하는 전 세계적으로 통용되는 표준 코드
      - 제품 포장재에서 흔히 볼 수 있는 13자리(EAN-13) 또는 12자리(UPC) 숫자로 구성
    - 목적 
      - 유통 채널 전반에서 상품을 고유하게 식별
      - 특히 오프라인 소매점 및 대규모 물류 시스템에서 필수
-  플랫폼 종속적 상품 코드 (Platform-Specific ID)
    - 정의 
      - 쿠팡의 `productId`, 네이버의 `nv_mid`, G마켓의 `EPIN`처럼 각 쇼핑몰 플랫폼이 내부적으로 상품을 관리하고 검색 결과를 구성하기 위해 사용하는 코드
    - 목적 
      - 플랫폼 내 가격 비교, 검색 랭킹, 광고 집행 등의 기준이 됩니다 판매자가 직접 제어하기보다는 플랫폼의 정책에 따라 생성되고 관리

## 좋은 상품 코드(SKU) 설계의 5가지 원칙
    1. 고유성 (Uniqueness)  
       - 하나의 코드는 오직 하나의 상품 단위(SKU)만을 가리켜야 함
       - 중복은 데이터 오염의 시작이며, 모든 재고 및 판매 데이터의 신뢰도를 무너뜨림
    2. 확장성 (Scalability)  
       - 미래에 새로운 카테고리, 브랜드, 속성이 추가되더라도 코드 체계가 무너지지 않고 유연하게 대응할 수 있음
    3. 직관성 (Intuitiveness)  
       - 코드의 일부만 보아도 상품의 기본 정보(예: 카테고리, 브랜드)를 유추 (예: `TSH-NKE-001`은 나이키 티셔츠)
    4. 일관성 (Consistency)  
       - 모든 상품에 예외 없이 동일한 구조와 명명 규칙이 적용 
       - 규칙의 일관성은 시스템의 안정성을 보장
    5. 시스템 친화성 (System-Friendliness)  
       - 시스템이 처리하기 어려운 특수문자(\*, /, % 등) 안씀
       - 혼동하기 쉬운 문자(알파벳 O와 숫자 0, 알파벳 I와 숫자 1) 안씀
       - 엑셀 등에서 날짜로 자동 변환될 수 있는 형식도 피함

## 주요 온라인 마켓 코드 시스템 비교
### 1. 쿠팡 — 아이템 마켓 중심의 중앙집권형 코드 구조
#### 핵심 코드 체계
   - productId (노출상품ID)  
     - 고객에게 보이는 최종 상품 페이지 ID  
     - 동일 상품(여러 판매자) → 하나의 `productId`로 묶임
   - sellerProductId (등록상품ID)  
     - 판매자 단위 상품 등록 시 생성  
     - 판매자 내부 관리 기준
   - vendorItemId (옵션ID)  
     - 옵션 단위 ID(색상, 사이즈 등)  
     - 가격·재고는 이 단위에서 실제 관리됨
#### 구조적 특징
   - 숫자 기반 비식별 코드 → 빠른 조회·인덱싱 목적
   - 등록 상품과 노출 상품 분리 → 쿠팡이 카탈로그를 직접 통제
   - 판매자 자율성 최소화 → 플랫폼 중심의 통합 데이터 관리

### 2. 네이버 스마트스토어 — 검색 중심 + 판매자 자율성 공존 구조
#### 핵심 코드 체계
   - nv_mid (네이버 쇼핑 상품 ID)  
     - 검색·가격비교에서 상품을 식별  
     - 동일 상품 묶기 기준
   - 판매자 상품코드  
     - 판매자가 자유롭게 설정  
     - ERP·SKU 연동 가능
   - 스토어 상품코드  
     - 스마트스토어 내부 고유 ID  
     - `nv_mid`와 별도 관리
#### 구조적 특징
   - 플랫폼 검색용 `nv_mid` + 판매자용 상품코드 → 이중 구조  
   - 검색 최적화가 중심  
   - 판매자 시스템 연동 가능 → 자율성 높음

### 3. G마켓·옥션 (ESM+) — 정형 데이터 기반의 카탈로그형 구조
#### 핵심 코드 체계
   - EPIN (상품 모델 식별 코드)  
     - 카테고리·속성·모델명·바코드 기반  
     - 하나의 EPIN → 실제 시장에 존재하는 ‘한 가지 모델’ 정의
   - 판매자관리코드  
     - 판매자가 내부 용도로 자유 사용
#### 구조적 특징
   - 데이터 정합성 중심 → 중복 상품 통합  
   - 바코드·모델명 기반으로 정확한 매칭  
   - 의류·자체제작 상품은 매칭 어렵고 자율성 낮을 수 있음

## 플랫폼별 특징 비교 표

| 항목 | 쿠팡 | 네이버 스마트스토어 | G마켓·옥션 (ESM+) |
|------|------|------------------------|-------------------------|
| 코드 구조 성격 | 중앙집권형 | 검색 중심 + 자율성 | 카탈로그 기반 구조화 |
| 주요 코드 | productId / sellerProductId / vendorItemId | nv_mid / 판매자코드 / 스토어코드 | EPIN / 판매자관리코드 |
| 코드 의미 | 플랫폼 중심으로 상품 통합 관리 | 검색용 ID + 판매자 SKU 병행 | 표준화된 모델 기반 ID 생성 |
| 판매자 자율성 | 매우 낮음 | 매우 높음 | 중간(상품 종류 따라 제약 있음) |
| 상품 통합 방식 | 쿠팡이 직접 카탈로그 운영 | 검색엔진 기준으로 묶임 | 모델명·바코드 기반 통합 |
| 장점 | 빠른 조회, 일관된 카탈로그 | 자유로운 ERP 연동 | 정확한 모델 매칭, 데이터 정합성 |
| 단점 | 판매자 유연성 부족 | 상품 통합은 플랫폼 기준 | 의류 등 EPIN 정의 어려움 |

## 레드포스트 상품 코드(SKU) 설계 가이드
### Step 1. 핵심 목표 및 설계 원칙 정의
- 관리 목표 설정 (예시)
    - [재고 관리 효율성 향상]: 사이즈/색상별 재고를 실시간으로 정확하게 파악
    - [데이터 분석 정확도 증대]: 상품별/카테고리별 매출 및 수익성 분석
    - [운영 자동화 기반 마련]: 주문 수집, 발주, 송장 처리 자동화의 기준점 마련
- 설계 4대 원칙 확립
    - 고유성(Uniqueness): 모든 SKU는 시스템 내에서 유일한 값을 가진다
    - 확장성(Scalability): 신규 카테고리, 브랜드, 상품이 추가되어도 규칙을 수정할 필요가 없다
    - 직관성(Intuitiveness): 코드의 일부만으로 카테고리, 브랜드 등 기본 정보를 추측할 수 있도록 한다
    - 일관성(Consistency): 모든 상품에 동일한 구조와 생성 규칙을 적용한다
### Step 2. 코드 구성 요소 및 형식 결정

상품카테고리

#### 옵션(색상, 사이즈)은 어떻게?
  -  옵션은 기본 SKU에 추가 코드를 붙이는 방식으로 관리하여 기본 상품 정보와 옵션 정보를 명확히 분리
      - 예시: `TOP-BR1-SL-2510-0001`(기본 상품)
      - 옵션 추가: `TOP-BR1-SL-2510-0001`-RD-100 (빨강 + L 사이즈)

### Step 3 코드 생성 및 관리 규칙 확정

결정된 구성 요소를 바탕으로 구체적인 운영 규칙을 문서화

- 최종 코드 조합 형식: `[카테고리][일련번호][옵션1]-[옵션2]`
- 생성 규칙:
    - 규칙 1: 모든 코드는 사전에 정의된 카테고리'마스터 데이터 코드표'를 따른다 
    - 규칙 2: 일련번호는 4자리
    - 규칙 3: 특수 상품(예: 세트, 사은품)은 카테고리 코드 자리에 별도 접두사(예: SET, GFT)를 사용한다
    - 규칙 4: 한번 부여된 SKU는 상품이 단종되더라도 절대 재사용하지 않는다 (과거 판매 데이터와의 연결성 유지)

### Step 4 시스템 적용 및 검증

설계된 규칙을 실제 상품에 적용해보고 문제점이 없는지 검증

- Case 1: 일반 상품 등록
    - 상품 정보: 의류 > 하의 > 청바지 / '데님메이커' 브랜드 / 스트레이트핏 / 25년 11월 등록 / 5번째 상품
    - 생성 코드: `BOT-DM1-ST-2511-0005`
    - 옵션 (허리 30인치): `BOT-DM1-ST-2511-0005-30`
- Case 2: 세트 상품 등록
    - 상품 정보: 세트 상품 / '홈웨어' 브랜드 / 25년 12월 등록 / 12번째 상품
    - 생성 코드: `SET-HW1-NA-2512-0012` (속성이 불필요할 경우 NA 등으로 표기)

# 결론: 상품 코드는 살아있는 비즈니스 언어

상품 코드는 단순한 번호가 아닌, 비즈니스의 전략과 철학이 담긴 '설계의 결과물'입니다

성공적인 자체 상품 코드 시스템 구축은 단기적으로는 재고 관리의 정확성을 높이고 운영 비용을 절감합니다 장기적으로는 정확한 데이터 분석을 통해 어떤 상품을 더 소싱하고, 어떤 프로모션을 진행해야 할지 알려주는 전략적 나침반이 될 것입니다

체계적으로 쌓인 데이터는 미래에 그 어떤 자산보다 강력한 경쟁력이 되어 돌아올 것입니다
