# 📦 (중국 창고)입고관리

## 1. 입고 프로세스 (Workflow)

중국 창고에 화물이 도착했을 때의 처리 절차입니다.

### 1-1. 트래킹 번호 스캔 및 대조
1.  **스캔**: 핸드 스캐너 또는 모바일 앱을 사용하여 도착한 화물의 **트래킹 번호(운송장 번호) 바코드**를 스캔합니다.
2.  **데이터 대조**: 스캔 된 번호를 `고객 주문 DB`와 실시간 비교합니다.
    * **매칭 성공**: [정상 입고] 프로세스로 진행
    * **매칭 실패**: [노데이터(미신청)] 프로세스로 이동

### 1-2. 정상 입고 처리
1.  **확인**: 시스템에서 고객의 주문 내역이 확인됩니다.
2.  **QR 라벨 발급**: `주문 DB` 정보를 바탕으로 내부 관리용 **QR 라벨**을 즉시 출력합니다.
    * **QR 포함 정보**: 고객 주문번호, 사서함 번호(고객 식별자), 트래킹 번호, 상품 사진 링크
3.  **라벨 부착**: 출력된 QR 라벨을 박스 외부에 택배 송장처럼 부착합니다.
4.  **효과**:
    * 박스를 개봉하지 않아도 QR 스캔만으로 어떤 고객의 물건인지 식별 가능.
    * 합배송 등으로 여러 고객의 물건이 섞여 있어도 스캔 한 번으로 즉시 분류 가능.
    * 검수 및 출고 단계에서 처리 속도 향상.

### 1-3. 노데이터 (미신청 화물) 처리
* **정의**: 트래킹 번호가 시스템상 어떠한 고객 주문과도 매칭되지 않는 경우.
* **처리 프로세스**:
    1.  시스템에서 '노데이터'로 자동 분류.
    2.  관리자가 해당 박스의 외관 및 운송장 사진을 촬영하여 시스템에 업로드.
    3.  추후 고객이 트래킹 번호를 입력하면 매칭되도록 대기 상태 유지.

## 2. 데이터 구조 (Data Structure)

### 2-1. 엑셀(Excel) 연동 데이터 포맷
스캐너가 QR을 읽어 엑셀이나 시스템에 전송할 때 사용하는 문자열 형식 예시입니다.

| 항목 | 예시 데이터 | 설명 |
| :--- | :--- | :--- |
| **주문번호** | `250203-B0001-0099` | 고유 주문 번호 |
| **사서함번호** | `RP0099` | 고객 식별 번호 |
| **상품코드** | `CLO1010-0052-B160` | 상품정보 (옵션: 색상/사이즈 등) |

### 2-2. 데이터베이스 (DB) 설계안
입고 관리를 위해 필요한 주요 필드 및 상태 값입니다.

#### A. 주요 필드
* `tracking_number` (PK): 트래킹 번호 (운송장 번호)
* `order_number`: 주문 번호 (매칭 시)
* `customer_id`: 고객 번호 (사서함 번호)
* `inbound_date`: 입고 일자 (보관료 산정 기준일)
* `image_url`: 노데이터 시 업로드한 상자/운송장 사진 경로

#### B. 입고 상태 (Status Code)
* **입고 전 (Pending)**: 데이터는 있지만 물건 미도착
* **입고 완료 (Received)**:
    * `NORMAL`: 정상 입고 (주문서 매칭 완료)
    * `NODATA`: 노데이터 (주문서 미매칭)

---

## 3. 보관료 정책 (Storage Fee Policy)

`입고 일자`를 기준으로 경과 기간을 계산하여 상태를 분류합니다.

| 경과 기간 | 상태 구분 | 비고 |
| :--- | :--- | :--- |
| **0일 ~ 7일** | **무료 보관** | 기본 서비스 구간 |
| **8일 ~ 30일** | **장기 보관 (방치)** | 알림 발송 필요 |
| **31일 이후~** | **유료 보관** | 보관료 청구 또는 폐기 검토 |
